If you sell for a living, you have probably used a LinkedIn automation tool. Expandi, Heyreach, Linked Helper, Dripify.

They all send connection requests on your behalf. Almost none of them explain what's actually happening when they do it.

This post follows a single connection request from the moment you'd click Connect, all the way down to the one thing LinkedIn's computers actually receive. Plain English first. The tech names come at the end.

## The human version

What you do yourself:

- Open the prospect's LinkedIn profile.
- Click the blue **Connect** button.
- A box pops up asking if you want to add a note.
- Type a short message and hit **Send**.
- They get a notification. If they accept, you're connected.

That's the visible part. The invisible part is what automation has to copy.

## What's actually going on when you click Send

Pressing Send doesn't really "click" anything. Your browser quietly sends a short, structured message to LinkedIn's computers. It says, in effect:

*"Create a new invitation from me to this person, with this note."*

LinkedIn's computers reply: *"done"* or *"no, here's why not."* Your screen updates based on the reply.

Three things travel with that message every time:

- **An address.** A specific spot inside LinkedIn that handles invitations. Only one such spot exists. Every invite, from anyone, goes through it.
- **A small package of info.** Who the invite is for, and what note to include.
- **Proof it's really you.** A bundle of identification your browser carries, so LinkedIn knows the request is from your account.

If any of those three are missing or wrong, the invite fails.

## What automation actually does

An automation tool does the same three things, without you clicking.

The recipe:

- Get a copy of your login credentials from your browser, one time, by hand.
- Look up the prospect's internal ID inside LinkedIn.
- Build the same message a browser would build: prospect, note, login proof.
- Send it to that one address.
- Read the reply. If it worked, log it. If it didn't, react.

That's the whole trick. *Build the same request your browser would build, then send it without the browser.*

## What LinkedIn says back

You never see these replies yourself. The tool sees them and reacts.

- **"OK, done."** Invite sent. Counter ticks up.
- **"You've hit your weekly limit."** LinkedIn caps invites somewhere around 100 to 200 per account per week. The cap moves around without warning.
- **"You look like a bot."** The account needs a cool-down period before it can send more.
- **"Already connected"** or **"already invited."** Skip the prospect.

Those four cases cover almost every outcome. A good tool handles each one quietly so you don't have to.

## Why automation tools sometimes open a real browser anyway

If the recipe is so simple, why do some tools run a hidden browser in the background?

- **LinkedIn occasionally shows an anti-bot challenge on this endpoint.** Solving it needs a real browser, because the challenge runs as code inside the page.
- **Faking the login proof by hand is fiddly.** A real browser builds it all for you, so the tool doesn't have to maintain a perfect impersonation.

For the actual invite, though, the request itself is the small structured message described above. No browser strictly required, as long as the impersonation holds up.

## The picture in three lines

- A connection request is a small message sent from your browser to one specific spot inside LinkedIn.
- Automation is software that sends the same message without you clicking.
- Everything LinkedIn checks (login proof, prospect ID, note, weekly cap) gets checked at that one spot.

## Now, the technical names

If you want to read a tool's docs or talk to your engineer about this without getting lost:

| Plain English | Technical name | What it actually is |
| --- | --- | --- |
| The short message your browser sends | **HTTP request** | The standard way browsers and servers talk. |
| LinkedIn's behind-the-scenes URLs | **Voyager API** | LinkedIn's internal API. Same one their mobile app uses. URLs start with `linkedin.com/voyager/api/...`. |
| The one address that handles invites | `voyagerRelationshipsDashMemberRelationships?action=verifyQuotaAndCreateV2` | The single URL every connection request goes to, human or automated. |
| The prospect's internal LinkedIn ID | **URN** (Uniform Resource Name) | Looks like `urn:li:fsd_profile:ACoAAB1234...`. The URL slug (`/in/jane-doe`) is for humans; the URN is what LinkedIn's code uses. |
| Your login credentials | **`li_at` cookie** | A short text token your browser carries. Anyone holding a valid `li_at` can act as that account. |
| The extra security code | **CSRF token** | Cross-Site Request Forgery token. Stops random websites tricking your browser into firing invites for them. |
| The "I was on Jane's profile" stamp | **`x-li-page-instance`** | A header that tells LinkedIn the click came from a sensible page. |
| The reply codes | **HTTP status codes** | `200` = OK. `429` = weekly cap hit. `403` = looks like a bot. The tool reads these and reacts. |

That's the whole vocabulary you need for this one endpoint. Every connection-request automation, on every tool, is built out of those pieces.
